# Разработка виджетов — быстрый старт

Вам дали zip этого репозитория. Вот что нужно сделать, чтобы запуститься и
начать разрабатывать виджеты локально.

---

## 1. Быстрый старт

### Вариант A — Docker-образ (рекомендуется)

Готовый образ уже содержит правильно установленный `node_modules` —
Node.js, npm, `.npmrc`/Nexus-токен и `--legacy-peer-deps` вам не нужны
вообще, независимо от того, откуда вы получили проект.

**Требования:** Docker (Desktop на Windows/Mac, Engine на Linux). Для
работы с ИИ в IDE — Cursor или VS Code с расширением **Dev Containers**.

1. Распакуйте zip / склонируйте репозиторий, откройте папку в Cursor.
2. Cursor покажет всплывающее уведомление **«Reopen in Container»** —
   нажмите его (или `Ctrl/Cmd+Shift+P` → `Dev Containers: Reopen in
   Container`).
3. Первый раз Docker скачает готовый образ (`docker-compose.yml` тянет
   `ghcr.io/app-insight/insight-widgets:latest`) — заметно быстрее, чем
   собирать самому, потому что зависимости уже внутри.
4. Контейнер поднимется — dev-сервер (`npm run start`) уже работает как
   основной процесс контейнера, порт `3001` Cursor прокинет на
   `localhost` автоматически.

Без Cursor — то же самое одной командой:
```
docker compose up
```

Готово — можно разрабатывать. Дальше подключаете виджет к редактору
(раздел 2) и пишете код (раздел 4). Как публикуется сам образ — раздел 6.

### Вариант B — локально, без Docker

**Требования:** Node.js ≥ 18, npm ≥ 9 (`node -v`, `npm -v`), доступ к
npm-реестру Nexus (`art.goodt.me`) — registry и публичный токен уже
прописаны в `.npmrc` репозитория.

1. Распакуйте zip в рабочую папку и откройте её в IDE (VS Code / Cursor /
   WebStorm — неважно, дальше всё через терминал).

2. Установите зависимости **с флагом `--legacy-peer-deps`**:
   ```
   npm install --legacy-peer-deps
   ```
   Флаг обязателен: `peerDependencies` пакета (`goodt-wcore`, `vue` и т.д.)
   местами пересекаются/устарели, и без флага npm откажется ставить пакеты
   с ошибкой `ERESOLVE`. Запоминайте флаг — он нужен и при `npm install`
   отдельных пакетов, и при `npm up` в будущем.

3. Синхронизируйте версии зависимостей с текущей версией платформы
   (тянет актуальные версии `goodt-*` пакетов под `vendorOptions.platformUrl`
   из `package.json`):
   ```
   npm run update:platform-deps
   npm up --force --legacy-peer-deps
   ```

4. Запустите dev-сервер:
   ```
   npm run start
   ```
   Поднимется на `http://localhost:3001` (порт можно поменять переменной
   окружения `DEV_SERVER_PORT`).

Готово — можно разрабатывать. Дальше подключаете виджет к редактору
(раздел 2) и пишете код (раздел 4).

## 2. Подключение виджета в редакторе (`$elem.add`)

Пока dev-сервер запущен, откройте страницу редактора платформы (адрес — в
`package.json` → `vendorOptions.platformUrl`, например
`https://insight-release-test.goodt.me/editor/`), откройте консоль браузера
(F12 → Console) и выполните:

```js
$elem.add({
    types: ['ElemCalendar', 'ElemAccordion'],
    url: 'http://192.168.10.172:3001'
});
```

- `types` — имена виджетов (папок в `./src`), которые нужно подключить.
- `url` — адрес вашего dev-сервера. Если браузер и dev-сервер на одной
  машине — можно `http://localhost:3001`. Если редактор открыт с другого
  устройства (или его открывает коллега/клиент) — укажите IP вашей машины в
  локальной сети, как в примере выше.

После этого указанные виджеты появятся в палитре редактора и будут
подтягиваться прямо с вашего локального сервера — билд и деплой не нужны.

Команда `$elem.remove()` откатывает подключение (нужно обновить страницу).

**Как это работает.** `$elem` — глобальный объект, который сама страница
редактора создаёт при загрузке (это видно в собранном бандле редактора,
см. `редактор/editor.5973.*.js` в этом репозитории). Он хранит `{types,
url}` в `localStorage` под ключом рантайм-пакета, поэтому подключение
переживает обновление страницы, пока вы явно не вызовете `$elem.remove()`.
Официально Goodt эту возможность не документирует — это то же самое, что
раньше делалось через `npm run serve:platform` + `platform.local.json`
(см. старый процесс в конце файла), только без поднятия прокси-сервера.

## 3. Простой деплой виджетов (опционально)

`$elem.add` — быстрый способ проверить виджет прямо сейчас, но подключение
живёт только в вашем браузере (через `localStorage`) и слетает, если
вызвать `$elem.remove()` или почистить браузер. Когда виджет готов и его
нужно стабильно выкатить на сервер редактора (Docker), используйте этот
процесс.

**Итоговая структура на сервере:**

```
insight/
├── docker-compose-app.yaml
└── widgets.local/
    ├── config/
    │   └── widgets.local.json       # регистрация пакета
    └── {your_name}/                 # содержимое ./dist
        ├── ...js файлы виджетов...
        └── config/package/
            └── widgets.local.json   # виджеты в панели
```

**Шаг 1.** Соберите виджеты на машине разработчика:
```
npm run build
```
Результат появится в папке `./dist`.

**Шаг 2.** На сервере создайте папки рядом с `docker-compose-app.yaml`:
```
mkdir -p ./widgets.local/config
mkdir -p ./widgets.local/{your_name}
```

**Шаг 3.** Скопируйте содержимое `./dist` в папку пакета:
```
cp -r ./dist/* ./widgets.local/{your_name}/
```

**Шаг 4.** Проверьте список виджетов в панели — после копирования
проверьте и при необходимости отредактируйте файл
`./widgets.local/{your_name}/config/package/widgets.local.json`. В нём
описаны виджеты, которые появятся в панели редактора:
```json
{
  "elements": [
    {
      "type": "{your_name}::ElemHelloWorld",
      "name": "Hello World",
      "icon": "mdi-earth"
    }
  ],
  "tools": {
    "page": [
      { "name": "Vendor", "elements": ["{your_name}::ElemHelloWorld"] }
    ],
    "template": [
      { "name": "Vendor", "elements": ["{your_name}::ElemHelloWorld"] }
    ]
  }
}
```

**Шаг 5.** Создайте файл `./widgets.local/config/widgets.local.json`:
```json
{
  "widgets": {
    "packages": {
      "{your_name}": { "url": "./widgets.local/{your_name}" }
    }
  }
}
```

**Шаг 6.** Добавьте volume в `docker-compose-app.yaml` в секцию
`ins-editor`:
```
volumes:
  - ./widgets.local:/abc/app/goodt/editor/widgets.local
```

**Шаг 7.** Перезапустите контейнер:
```
docker compose restart ins-editor
```

**Шаг 8.** Виджеты появятся в панели редактора в группе, указанной в
`tools.page[0].name` внутри `./dist/config/package/widgets.local.json`.

При обновлении виджетов — повторить шаги 1, 3 и 6.

> Шаги 2, 5 и 6 (папки, `widgets.local/config/widgets.local.json`, volume
> в `docker-compose-app.yaml`) — разовая настройка сервера. При каждом
> следующем обновлении виджетов нужны собственно пересборка и копирование
> (шаги 1 и 3); проверьте у себя, нужен ли реально повторный шаг 6
> (volume не меняется между обновлениями) или же здесь имелся в виду
> перезапуск контейнера (шаг 7) — уточните на своей стороне.

## 4. Разработка с ИИ

В репозитории уже есть контекст, которым ИИ-агент (Claude Code, Cursor и
т.п.) может пользоваться, чтобы писать код виджетов в едином стиле:

- **`.cursor/rules/widget.mdc`** — плейбук по структуре виджета
  (`descriptor.js`, `panels/`, `Elem<Name>.vue`, `utils/`, `styles/`) с
  примерами на реальных виджетах пакета.
- **`WIDGET_REFACTORING_SPEC.md`** — стандарт панелей «Оформление» /
  «Я дизайнер», интеграция с Vuex, готовые шаблоны кода.
- Сами виджеты в `./src/Elem*` — лучшие референсы для конкретных паттернов.

Типичный цикл:

1. Опишите агенту задачу («создай виджет ElemXxx по образцу ElemYyy» /
   «поправь панель Оформление у ElemXxx»).
2. Пока агент пишет код в `./src`, dev-сервер (запущенный в разделе 1) сам
   пересобирает изменённые файлы — просто обновите страницу редактора.
3. Если виджет новый — подключите его через `$elem.add` (раздел 2) для
   быстрой проверки, а когда он готов — задеплойте на сервер (раздел 3).

`npm run create` (обёртка над `goodt-scaffold`) по-прежнему доступен для
генерации заготовки виджета, если нужен быстрый скелет без ИИ.

## 5. Траблшутинг

| Проблема | Решение |
|---|---|
| `npm install` падает с `ERESOLVE` / конфликтом peer dependencies | Добавьте флаг: `npm install --legacy-peer-deps`. Это ожидаемо для этого пакета, см. раздел 1. Либо используйте вариант A (Docker) — там этот шаг не нужен вовсе. |
| `401 Unauthorized` при `npm install` | Токен для `art.goodt.me` не задан/истёк — проверьте `.npmrc` (registry + `_authToken` уже должны быть прописаны в репозитории). |
| `update:platform-deps` падает с `SyntaxError: Unexpected token '<'` | `platformUrl` в `package.json` недоступен из вашей сети или указан неверно — откройте его в браузере и проверьте. |
| `ReferenceError: fetch is not defined` | Node.js < 18 — обновите Node.js (или используйте вариант A). |
| Порт `3001` занят | `DEV_SERVER_PORT=3002 npm run start` (вариант B) или `WIDGETS_IMAGE=... DEV_SERVER_PORT=3002 docker compose up` / поменяйте `3001:3001` на `3002:3001` в `docker-compose.yml` (вариант A). |
| `$elem` не определён в консоли | Обновите страницу редактора — глобальный объект создаётся при загрузке бандла редактора; либо страница открыта не из этой сборки редактора. |
| Виджет не появился после `$elem.add` | Проверьте, что `types` совпадают с именами папок в `./src` (например, `ElemAccordion`, а не `elem-accordion`), и что `url` доступен из браузера (не заблокирован файрволом/не `localhost`, если браузер на другой машине). |
| После правок в `./src` изменений не видно | Убедитесь, что dev-сервер ещё запущен и не упал с ошибкой сборки — смотрите его консоль (в Cursor — интегрированный терминал/лог Dev Containers; без него — `docker compose logs -f`). |
| `Module not found: Can't resolve '@goodt-widgets-insight/...'` | Виджет ссылается на локальный пакет, не объявленный в `package.json` (в отличие от `@goodt-widgets-insight/utils`, объявленного как `file:./src/common/utils`). Добавьте такую же `file:`-зависимость либо поправьте импорт. (`ElemPivotTable` с этой проблемой был удалён из репозитория как демо-виджет.) |

## 6. Раздача Docker-образа клиентам

Образ, который тянет `docker-compose.yml` (вариант A из раздела 1), нужно
один раз собрать на машине **с доступом к Nexus** — например, на
Ubuntu-машине. Дальше клиенты получают уже готовый образ и никогда сами
не собирают его и не трогают Nexus.

Сейчас (пока нет реестра) — раздаём файлом. Позже, когда появится реестр
— публикацией (раздел 6.2), тогда файлом слать не нужно будет вовсе.

### 6.1 Передача файлом (текущий способ)

На машине сборки (с доступом к Nexus) — **один** файл для клиента:
```
./scripts/build-client-package.sh                       # → insight-widgets-client-package.zip
./scripts/build-client-package.sh my-package.zip         # своё имя файла
```
Скрипт (если `insight-widgets.tar.gz` ещё не собран — сначала вызывает
`build-and-save-image.sh`, см. ниже), затем: экспортирует проект через
`git archive HEAD` (только закоммиченные файлы — без `node_modules`,
`dist`, `.git`) и кладёт `insight-widgets.tar.gz` прямо внутрь, рядом с
`docker-compose.yml`. Результат — один zip.

Клиенту нужно: поставить Docker Desktop и Cursor — и всё. Дальше:
1. Распаковать присланный zip.
2. Открыть получившуюся папку в Cursor.
3. Нажать всплывающее уведомление **«Reopen in Container»**.

На этом всё — никаких команд руками. `.devcontainer/devcontainer.json`
содержит `initializeCommand`, который сам выполняет `docker load -i
insight-widgets.tar.gz` на шаге «Reopen in Container» (до того, как
контейнер начнёт собираться/подниматься), поэтому образ подхватывается
автоматически. Команда некритичная (`|| exit 0`) — если файла нет (у
разработчика, открывающего сам репозиторий без прикреплённого образа),
просто ничего не делает и не мешает обычной пересборке по
`docker-compose.build.yml`.

Проверено: `git archive` корректно экспортирует дерево проекта
(`.devcontainer/`, `docker-compose.yml` и т.д. на своих местах),
`insight-widgets.tar.gz` кладётся туда же — распакованная структура
именно такая, какая нужна `initializeCommand`.

Собрать сам образ (если ещё не собран — `build-client-package.sh` это
сделает сам, но можно и отдельно):
```
./scripts/build-and-save-image.sh                       # → insight-widgets.tar.gz
```
Проверено вживую: реальная сборка (`npm install --legacy-peer-deps` из
`art.goodt.me`) ставит 1877 пакетов и занимает около 4 минут при полной
пересборке (при неизменных зависимостях — секунды, слой из кэша).

Если клиенту всё же удобнее без Cursor, руками — можно и так (не
основной путь):
```
docker load -i insight-widgets.tar.gz
docker compose up
```
(команды выполняются из уже распакованной папки — там сразу и образ, и
`docker-compose.yml`, отдельно доставать проект из образа не нужно).

### 6.2 Публикация в реестр (на будущее)

Когда появится реестр для образов — публикация вместо ручной передачи
файла: не нужно будет пересылать `.tar.gz` при каждом обновлении, клиенты
просто получат новую версию через `docker compose pull`.

**Требования на машине сборки:** доступ к `art.goodt.me`, логин в целевой
реестр (по умолчанию в файлах — GitHub Container Registry, `ghcr.io`,
привязан к этому GitHub-репозиторию; легко поменять на свой):
```
echo "<GitHub PAT со scope write:packages>" | docker login ghcr.io -u <ваш GitHub-логин> --password-stdin
```

Сборка и публикация:
```
./scripts/build-and-push-image.sh          # тег latest
./scripts/build-and-push-image.sh v1.2.0   # + дополнительный версионный тег
```

Оба скрипта (`build-and-save-image.sh` и `build-and-push-image.sh`)
собирают ровно один и тот же образ через `docker-compose.build.yml` (это
override, добавляющий `build:` к тому же сервису, что описан в
`docker-compose.yml`) — разница только в том, что происходит с
результатом (файл vs push в реестр).

**Другой реестр** (Docker Hub, свой Nexus с Docker-репозиторием и т.д.) —
поменяйте значение по умолчанию в `docker-compose.yml`/
`docker-compose.build.yml` (`WIDGETS_IMAGE`) или передавайте его через
переменную окружения:
```
WIDGETS_IMAGE=docker.io/<org>/insight-widgets ./scripts/build-and-push-image.sh
```

**Когда пересобирать:** при изменении `package.json`/выходе новой версии
платформы — так же, как раньше `npm run update:platform-deps` + `npm up`,
только теперь это делает тот, кто пересобирает образ, один раз, а не
каждый клиент у себя.

## 7. Запуск без Node.js и без Docker (крайний случай)

Если недоступны ни Docker, ни Node.js (штатные варианты A и B из
раздела 1), в `tools/exe-launcher/` есть отдельный лаунчер
`widgets-launcher.exe`, который сам содержит рантайм Node.js и
распаковывает заранее собранный `node_modules.zip`. Это самый крайний
фолбэк, не основной путь. Подробности сборки — в
`tools/exe-launcher/build-sea.sh` и комментариях в
`tools/exe-launcher/start.js`.

## Старый процесс (для справки)

Ниже — предыдущая инструкция, актуальная, если нужен полноценный цикл

<details>
<summary>Развернуть старую инструкцию «Собственная разработка виджетов»</summary>

### Собственная разработка виджетов

#### Предварительные требования

1. Node.js версии не ниже 18.
2. npm версии не ниже 9.
3. Доступ к корпоративному репозиторию (Nexus) или публичному репозиторию, где хранятся необходимые npm-пакеты.
4. URL-адрес платформы, где развёрнут редактор (например, https://yourdomain.com/editor).

#### Инициализация проекта

**Шаг 2.1. Склонируйте шаблонный репозиторий**

Шаблон даёт базовую структуру и конфигурацию для разработки:

```
git clone ssh://git@gitlab.goodt.me:9022/goodt-shared/widgets/template.git
```

(Если у вас собственный репозиторий шаблона внутри компании — используйте соответствующий адрес.)

**Шаг 2.2. Настройте package.json**

Название пакета. В package.json замените поле name на нужное, например:

```json
{
  "name": "@goodt-widgets-package/my-custom-widgets",
  "version": "1.0.0"
}
```

Адрес редактора (platformUrl). В разделе vendorOptions укажите рабочий URL редактора:

```json
{
  "vendorOptions": {
    "widgetsDir": "./src",
    "platformUrl": "https://yourdomain.com/editor"
  }
}
```

Это обязательно для корректной синхронизации зависимостей (см. далее).

**Шаг 2.3. Настройка .npmrc (при необходимости)**

Если пакеты лежат во внутреннем репозитории Nexus, пропишите в .npmrc правильный registry и токен:

```
registry=https:///repository/npm-group-public/
```

```
npm config set "///repository/npm-group-public/:_authToken" "NpmToken.xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
```

Если используется публичный Nexus от Goodt:

```
npm config set "//art.goodt.me/repository/npm-group-public/:_authToken" "NpmToken.da6c3f79-972d-3af5-ac4e-3dcc9580e566"
```

**Шаг 2.4.** Установите зависимости с помощью команды `npm i`

**Шаг 2.5.** Синхронизируйте зависимости с платформой

```
npm run update:platform-deps
npm up --force
```

Скрипт update:platform-deps подтянет корректные версии нужных библиотек под текущую версию платформы. Команда npm up обновит версии пакетов с учётом изменений в package.json. Если при выполнении update:platform-deps возникает ошибка типа SyntaxError: Unexpected token '<', проверьте, корректно ли задан platformUrl и доступен ли сайт из вашей сети. Также убедитесь, что используете Node.js версии ≥ 18.

#### 3. Создание нового виджета

**Шаг 3.1. Генерация заготовки виджета**

В проекте уже есть встроенный скрипт: `npm run create`. Он предложит выбрать шаблон (например, Hello World) и задать имя виджета, например ElemHelloWorld. В папке ./src появится одноименная директория с базовыми файлами.

**Шаг 3.2. Регистрируем новый виджет**

Чтобы виджет появился в редакторе, нужно прописать его в ./public/config/package/widgets.local.json. Например:

```json
{
  "elements": [
    {
      "type": "my-custom-widgets::ElemHelloWorld",
      "name": "Hello World",
      "icon": "mdi-earth"
    }
  ],
  "tools": {
    "page": [
      {
        "name": "Пример группы",
        "elements": [
          "my-custom-widgets::ElemHelloWorld"
        ]
      }
    ],
    "template": [
      {
        "name": "Пример группы",
        "elements": [
          "my-custom-widgets::ElemHelloWorld"
        ]
      }
    ]
  }
}
```

Где: `type` — формат `<название_пакета>::Имя_Виджета`; `name` — название, которое увидит пользователь; `icon` — любая иконка из mdi.

#### 4. Локальная разработка и тестирование (Песочница)

**Шаг 4.1. Инициализация песочницы**: `npm run sandbox:init`. Создаст файлы в ./sandbox/config/ — в частности sandbox.local.json и widgets.local.json.

**Шаг 4.2.** Настройте widgets.local.json песочницы. Чтобы запустить ваш виджет в песочнице, опишите его там.

**Шаг 4.3. Запуск песочницы**: `npm start`. Приложение загрузится на локальном порту, например http://localhost:3001 или http://localhost:30xx. Откройте эту страницу и убедитесь, что виджет отображается.

#### 5. Проверка в реальном редакторе локально (через прокси)

Если вам нужно протестировать виджет прямо в редакторе, не дожидаясь выкладки на сервер, можно воспользоваться локальным прокси:

1. Запустите сборку в режиме локального веб-сервера: `npm run serve:webpack` (или иной скрипт, который поднимает локальный сервер на 3000 порту).
2. Настройте файл ./sandbox/config/platform.local.json. Укажите:
   - `platform.url` — адрес удалённого редактора (например, https://partner.goodt.me/editor/);
   - `widgets.vendor.url` — адрес вашего локального сервера (обычно http://127.0.0.1:3000/);
   - пропишите, какой именно widgets.local.json подменять, чтобы редактор читал ваши локальные настройки.
3. Запустите прокси сервер: `npm run serve:platform`. После этого редактор будет доступен, например, по адресу http://localhost:3030/editor, но уже подменять ваш пакет виджетов локальной версией.

Подробно см. файл platform-proxy.md.

#### 6. Сборка и деплой

**Шаг 6.1. Сборка.** Для формирования итоговых сборочных артефактов (JS-файлы, конфиг и пр.): `npm run build`. Итоговые файлы попадут в папку ./dist (по умолчанию). Если хотите изменить директорию, используйте: `npm run build --output-dir=./my-dist`.

**Шаг 6.2. Подготовка к выкладке.**

1. Убедитесь, что в widgets.local.json (папка ./public/config/package/) описаны все виджеты, которые вы хотите иметь в редакторе.
2. В корневом widgets.local.json (папка ./public/config/platform/) укажите путь до собираемого пакета:

```json
{
  "widgets": {
    "packages": {
      "my-custom-widgets": {
        "url": "./widgets.local/my-custom-widgets"
      }
    }
  }
}
```

**Шаг 6.3. Деплой на сервер.** Скопируйте содержимое ./dist (либо указанной при сборке директории) на сервер в папку `/widgets.local/my-custom-widgets/`, где my-custom-widgets — имя, совпадающее с @goodt-widgets-package/my-custom-widgets (без префиксов @goodt-widgets-package/). Задеплойте ./dist/config/platform/widgets.local.json в ./widgets.local/config/widgets.local.json (либо объедините с существующей конфигурацией).

В результате на сервере должна получиться структура вида:

```
editor/
player/
config/
widgets/
widgets.local/
    ├── config/
    │   └── widgets.local.json     // обновлённый конфиг
    ├── my-custom-widgets/
    │   └── ... файлы сборки ...
```

После этого, зайдя в редактор (обычно https://yourdomain.com/editor), вы увидите свой виджет в соответствующей группе, а при добавлении на страницу — сможете работать с ним так же, как с любым стандартным.

#### 7. Регулярное обновление зависимостей

Платформа может обновляться, и, чтобы ваши виджеты работали стабильно, нужно периодически: обновлять зависимости (`npm run update:platform-deps`, `npm up`), пересобирать пакет (`npm run build`), повторно деплоить полученные артефакты.

#### 8. Устранение неполадок

1. Скрипт update:platform-deps падает с ошибкой `SyntaxError: Unexpected token '<'` — проверьте, правильно ли указан platformUrl в package.json; убедитесь, что сервер доступен, и URL действительно открывается.
2. Ошибка `ReferenceError: fetch is not defined` — используется Node.js версии ниже 18, нужно обновить Node.js.
3. `401` при установке пакетов (`npm i` или `npm up`) — проверьте корректность токена Nexus или настроек .npmrc.

Полная документация: https://docs.goodt.me/front/#/widgets-package/guide
