# Быстрая разработка виджетов (упрощённый процесс)

Этот документ описывает **новый, упрощённый** способ локальной разработки
виджетов для этого пакета — без клонирования шаблона, без настройки
`.npmrc`/Nexus-токена, без `npm run update:platform-deps`, без песочницы и
без прокси редактора. Старый процесс (см. ниже, в конце файла) остаётся
рабочим, но требует доступа к корпоративному Nexus и намного больше шагов.

Новый процесс сводится к трём вещам:

1. **`node_modules.zip`** — заранее собранный, рабочий `node_modules` этого
   проекта (со всеми legacy-`peerDependencies`, уже разрешёнными). Кладётся
   рядом с проектом и просто распаковывается — `npm i` не нужен, и, как
   следствие, не нужен доступ к приватному Nexus-реестру и не всплывают
   конфликты peer dependencies.
2. **`widgets-launcher.exe`** — лаунчер, который распаковывает
   `node_modules.zip` (если ещё не распакован) и поднимает dev-сервер
   (`vue-cli-service serve`, порт `3001`) — без установленного на машине
   Node.js/npm.
3. **`$elem.add(...)`** — команда в консоли браузера (F12) на странице
   редактора платформы, которая на лету подключает виджет с вашего
   локального dev-сервера в редактор — без сборки и без деплоя.

Поверх этого пишет код ИИ-агент (Claude Code / Cursor) — в репозитории уже
есть материалы для этого, см. [раздел «Разработка с ИИ»](#разработка-с-ии).

## 1. Быстрый старт

### Вариант A — без Node.js на машине (через .exe)

1. Скопируйте в одну папку:
   - весь проект (этот репозиторий);
   - `node_modules.zip` (получить у ответственного за сборку зависимостей —
     см. [раздел 5](#5-как-обновить-node_moduleszip));
   - `widgets-launcher.exe` (см. [раздел 4](#4-сборка-widgets-launcherexe),
     как его получить/собрать).
2. Запустите `widgets-launcher.exe` двойным кликом (или из консоли —
   тогда будут видны логи).
3. При первом запуске лаунчер распакует `node_modules.zip` в `node_modules/`
   рядом с собой, затем поднимет dev-сервер. При следующих запусках
   распаковка пропускается, если `node_modules` уже на месте.
4. В консоли лаунчера появится готовая строка для вставки в браузер —
   см. [раздел 2](#2-подключение-виджета-в-редакторе-elemadd).

### Вариант B — если Node.js ≥ 18 уже установлен

1. Распакуйте `node_modules.zip` в корень проекта (в папку `node_modules`).
2. Запустите:

   ```
   npm run start
   ```

3. Dev-сервер поднимется на `http://localhost:3001` (порт можно изменить
   переменной окружения `DEV_SERVER_PORT`).

## 2. Подключение виджета в редакторе (`$elem.add`)

Пока dev-сервер запущен, откройте страницу редактора платформы (например,
`https://insight-release-test.goodt.me/editor/` — актуальный адрес указан в
`package.json` → `vendorOptions.platformUrl`), откройте консоль браузера
(F12 → Console) и выполните:

```js
$elem.add({
    types: ['ElemCalendar', 'ElemAccordion'],
    url: 'http://192.168.10.172:3001'
});
```

- `types` — имена виджетов (папок в `./src`), которые нужно подключить.
- `url` — адрес вашего dev-сервера. Если браузер и dev-сервер на одной
  машине — можно `http://localhost:3001`. Если редактор открыт с другого
  устройства (или его открывает коллега/клиент) — укажите IP вашей машины в
  локальной сети, как в примере выше.

После этого указанные виджеты появятся в палитре редактора и будут
подтягиваться прямо с вашего локального сервера — билд и деплой не нужны.

Команда `$elem.remove()` откатывает подключение (нужно обновить страницу).

**Как это работает.** `$elem` — глобальный объект, который сама страница
редактора создаёт при загрузке (это видно в собранном бандле редактора,
см. `редактор/editor.5973.*.js` в этом репозитории). Он хранит `{types,
url}` в `localStorage` под ключом рантайм-пакета, поэтому подключение
переживает обновление страницы, пока вы явно не вызовете `$elem.remove()`.
Официально Goodt эту возможность не документирует — это то же самое, что
раньше делалось через `npm run serve:platform` + `platform.local.json`
(см. старый процесс в конце файла), только без поднятия прокси-сервера.

## 3. Разработка с ИИ

В репозитории уже есть контекст, которым ИИ-агент (Claude Code, Cursor и
т.п.) может пользоваться, чтобы писать код виджетов в едином стиле:

- **`.cursor/rules/widget.mdc`** — плейбук по структуре виджета
  (`descriptor.js`, `panels/`, `Elem<Name>.vue`, `utils/`, `styles/`) с
  примерами на реальных виджетах пакета.
- **`WIDGET_REFACTORING_SPEC.md`** — стандарт панелей «Оформление» /
  «Я дизайнер», интеграция с Vuex, готовые шаблоны кода.
- Сами виджеты в `./src/Elem*` — лучшие референсы для конкретных паттернов.

Типичный цикл:

1. Опишите агенту задачу («создай виджет ElemXxx по образцу ElemYyy» /
   «поправь панель Оформление у ElemXxx»).
2. Пока агент пишет код в `./src`, dev-сервер (запущенный на шаге 1) сам
   пересобирает изменённые файлы — просто обновите страницу редактора.
3. Если виджет новый — добавьте его тип через `$elem.add` (раздел 2) либо
   зарегистрируйте по-старому в `./public/config/package/widgets.local.json`
   (для постоянной разработки, не для разовой проверки).
4. Когда виджет готов — обычная сборка и деплой (раздел 6 старого процесса
   ниже) уже не связаны с `$elem.add`, это чисто дев-инструмент.

`npm run create` (обёртка над `goodt-scaffold`) по-прежнему доступен для
генерации заготовки виджета, если нужен быстрый скелет без ИИ.

## 4. Сборка `widgets-launcher.exe`

Исходники лаунчера — `tools/exe-launcher/`. Это отдельный маленький
Node.js-скрипт (`start.js`), который:

- ищет `node_modules.zip` рядом с собой и распаковывает его в
  `node_modules/`, если папки ещё нет (`tar`/PowerShell на Windows,
  `unzip`/`tar` на Linux/macOS);
- запускает `node_modules/@vue/cli-service/bin/vue-cli-service.js serve`,
  используя **встроенный в exe** Node.js-рантайм (через `pkg`), — поэтому
  на целевой машине не нужен установленный Node.js/npm;
- печатает готовую команду `$elem.add(...)` для вставки в консоль.

Логика проверена юнит-тестами по сценариям: первый запуск (распаковка),
повторный запуск (пропуск распаковки), отсутствие `node_modules.zip`
(понятная ошибка).

Есть два способа сборки — оба дают на выходе `dist/widgets-launcher-win.exe`.

### Способ 1 (рекомендуемый): Node.js Single Executable Application

```bash
cd tools/exe-launcher
./build-sea.sh          # или: npm run build:win:sea
```

Скрипт скачивает официальный Windows-билд Node.js с **nodejs.org** (не
GitHub) и «запекает» в его копию `start.js` через встроенный в сам Node.js
механизм [Single Executable Applications](https://nodejs.org/api/single-executable-applications.html)
(инструмент `postject`, ставится одноразово из обычного npm-реестра). Так
собран и проверен `widgets-launcher.exe` из этой поставки: SEA-blob
сгенерирован из `start.js`, инъекция в `node.exe` (v22.22.2, win-x64)
подтверждена — метка `NODE_SEA_FUSE_...:1` присутствует в бинарнике.
Полный прогон на реальной Windows-машине (двойной клик) я в текущем
окружении сделать не могу — нет доступа к Windows/Wine-рантайму, — поэтому
перед раскаткой на клиента стоит один раз проверить это руками.

> **Предупреждение Windows при первом запуске.** Официальный `node.exe`
> подписан Node.js Foundation; после инъекции blob'а эта подпись
> становится невалидной, поэтому SmartScreen может показать «Windows
> защитил ваш компьютер» при первом запуске — жмите «Подробнее» → «Всё
> равно выполнить». Чтобы убрать это предупреждение для конечных
> пользователей, бинарник нужно подписать своим code signing сертификатом
> (`signtool`).

### Способ 2: pkg

```bash
cd tools/exe-launcher
npm install
npm run build:win     # → dist/widgets-launcher-win.exe
```

Есть также `build:linux` и `build:macos` для соответствующих платформ и
`build:all` для всех трёх сразу.

> **Важно:** `npm run build:win` при первом запуске скачивает готовый
> Node.js-рантайм с GitHub Releases проекта `pkg` (~50 МБ, один раз,
> кешируется). Этот способ требует, чтобы GitHub Releases был доступен без
> ограничений — в изолированных окружениях с политикой «только свой
> репозиторий» (как в сессиях этого ИИ-агента) он не работает: попытки
> скачать бинарник и собрать Node.js из исходников как fallback обе
> провалились (403 от GitHub и несовместимость компилятора при сборке из
> исходников соответственно). Поэтому для сборки в таких окружениях
> используйте способ 1 (SEA) — он ходит только на nodejs.org и npm,
> которые обычно разрешены.

## 5. Как обновить `node_modules.zip`

`node_modules.zip` — не часть репозитория (слишком большой и содержит
разрешённые зависимости из приватного Nexus). Пересобирается тем, у кого
есть доступ к `art.goodt.me` (см. `.npmrc`), при изменении
`package.json`/выходе новой версии платформы:

```bash
npm i
npm run update:platform-deps
npm up --force
zip -r node_modules.zip node_modules
```

Готовый `node_modules.zip` кладётся рядом с проектом/лаунчером для всех, у
кого нет доступа к Nexus.

## 6. Траблшутинг

| Проблема | Решение |
|---|---|
| Лаунчер пишет «`node_modules.zip` не найден» | Положите `node_modules.zip` в ту же папку, где лежит `widgets-launcher.exe` (или откуда запускаете `npm run start`). |
| Порт `3001` занят | Запустите с `DEV_SERVER_PORT=3002 widgets-launcher.exe` (или `set DEV_SERVER_PORT=3002` в Windows-консоли перед запуском). |
| `$elem` не определён в консоли | Обновите страницу редактора — глобальный объект создаётся при загрузке бандла редактора; либо страница открыта не из этой сборки редактора. |
| Виджет не появился после `$elem.add` | Проверьте, что `types` совпадают с именами папок в `./src` (например, `ElemAccordion`, а не `elem-accordion`), и что `url` доступен из браузера (не заблокирован файрволом/не `localhost`, если браузер на другой машине). |
| После правок в `./src` изменений не видно | Убедитесь, что dev-сервер (лаунчер / `npm run start`) ещё запущен и не упал с ошибкой сборки — смотрите его консоль. |

## Старый процесс (для справки)

Ниже — предыдущая инструкция, актуальная, если нужен полноценный цикл
клонирования шаблона, работы с приватным Nexus и деплоя пакета на сервер
редактора (сборка, `widgets.local.json`, `./widgets.local/<package>/` на
сервере и т.д.). Для повседневной разработки и быстрой проверки виджета
предпочтителен процесс из разделов 1–3 выше.

<details>
<summary>Развернуть старую инструкцию «Собственная разработка виджетов»</summary>

### Собственная разработка виджетов

#### Предварительные требования

1. Node.js версии не ниже 18.
2. npm версии не ниже 9.
3. Доступ к корпоративному репозиторию (Nexus) или публичному репозиторию, где хранятся необходимые npm-пакеты.
4. URL-адрес платформы, где развёрнут редактор (например, https://yourdomain.com/editor).

#### Инициализация проекта

**Шаг 2.1. Склонируйте шаблонный репозиторий**

Шаблон даёт базовую структуру и конфигурацию для разработки:

```
git clone ssh://git@gitlab.goodt.me:9022/goodt-shared/widgets/template.git
```

(Если у вас собственный репозиторий шаблона внутри компании — используйте соответствующий адрес.)

**Шаг 2.2. Настройте package.json**

Название пакета. В package.json замените поле name на нужное, например:

```json
{
  "name": "@goodt-widgets-package/my-custom-widgets",
  "version": "1.0.0"
}
```

Адрес редактора (platformUrl). В разделе vendorOptions укажите рабочий URL редактора:

```json
{
  "vendorOptions": {
    "widgetsDir": "./src",
    "platformUrl": "https://yourdomain.com/editor"
  }
}
```

Это обязательно для корректной синхронизации зависимостей (см. далее).

**Шаг 2.3. Настройка .npmrc (при необходимости)**

Если пакеты лежат во внутреннем репозитории Nexus, пропишите в .npmrc правильный registry и токен:

```
registry=https:///repository/npm-group-public/
```

```
npm config set "///repository/npm-group-public/:_authToken" "NpmToken.xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
```

Если используется публичный Nexus от Goodt:

```
npm config set "//art.goodt.me/repository/npm-group-public/:_authToken" "NpmToken.da6c3f79-972d-3af5-ac4e-3dcc9580e566"
```

**Шаг 2.4.** Установите зависимости с помощью команды `npm i`

**Шаг 2.5.** Синхронизируйте зависимости с платформой

```
npm run update:platform-deps
npm up --force
```

Скрипт update:platform-deps подтянет корректные версии нужных библиотек под текущую версию платформы. Команда npm up обновит версии пакетов с учётом изменений в package.json. Если при выполнении update:platform-deps возникает ошибка типа SyntaxError: Unexpected token '<', проверьте, корректно ли задан platformUrl и доступен ли сайт из вашей сети. Также убедитесь, что используете Node.js версии ≥ 18.

#### 3. Создание нового виджета

**Шаг 3.1. Генерация заготовки виджета**

В проекте уже есть встроенный скрипт: `npm run create`. Он предложит выбрать шаблон (например, Hello World) и задать имя виджета, например ElemHelloWorld. В папке ./src появится одноименная директория с базовыми файлами.

**Шаг 3.2. Регистрируем новый виджет**

Чтобы виджет появился в редакторе, нужно прописать его в ./public/config/package/widgets.local.json. Например:

```json
{
  "elements": [
    {
      "type": "my-custom-widgets::ElemHelloWorld",
      "name": "Hello World",
      "icon": "mdi-earth"
    }
  ],
  "tools": {
    "page": [
      {
        "name": "Пример группы",
        "elements": [
          "my-custom-widgets::ElemHelloWorld"
        ]
      }
    ],
    "template": [
      {
        "name": "Пример группы",
        "elements": [
          "my-custom-widgets::ElemHelloWorld"
        ]
      }
    ]
  }
}
```

Где: `type` — формат `<название_пакета>::Имя_Виджета`; `name` — название, которое увидит пользователь; `icon` — любая иконка из mdi.

#### 4. Локальная разработка и тестирование (Песочница)

**Шаг 4.1. Инициализация песочницы**: `npm run sandbox:init`. Создаст файлы в ./sandbox/config/ — в частности sandbox.local.json и widgets.local.json.

**Шаг 4.2.** Настройте widgets.local.json песочницы. Чтобы запустить ваш виджет в песочнице, опишите его там.

**Шаг 4.3. Запуск песочницы**: `npm start`. Приложение загрузится на локальном порту, например http://localhost:3001 или http://localhost:30xx. Откройте эту страницу и убедитесь, что виджет отображается.

#### 5. Проверка в реальном редакторе локально (через прокси)

Если вам нужно протестировать виджет прямо в редакторе, не дожидаясь выкладки на сервер, можно воспользоваться локальным прокси:

1. Запустите сборку в режиме локального веб-сервера: `npm run serve:webpack` (или иной скрипт, который поднимает локальный сервер на 3000 порту).
2. Настройте файл ./sandbox/config/platform.local.json. Укажите:
   - `platform.url` — адрес удалённого редактора (например, https://partner.goodt.me/editor/);
   - `widgets.vendor.url` — адрес вашего локального сервера (обычно http://127.0.0.1:3000/);
   - пропишите, какой именно widgets.local.json подменять, чтобы редактор читал ваши локальные настройки.
3. Запустите прокси сервер: `npm run serve:platform`. После этого редактор будет доступен, например, по адресу http://localhost:3030/editor, но уже подменять ваш пакет виджетов локальной версией.

Подробно см. файл platform-proxy.md.

#### 6. Сборка и деплой

**Шаг 6.1. Сборка.** Для формирования итоговых сборочных артефактов (JS-файлы, конфиг и пр.): `npm run build`. Итоговые файлы попадут в папку ./dist (по умолчанию). Если хотите изменить директорию, используйте: `npm run build --output-dir=./my-dist`.

**Шаг 6.2. Подготовка к выкладке.**

1. Убедитесь, что в widgets.local.json (папка ./public/config/package/) описаны все виджеты, которые вы хотите иметь в редакторе.
2. В корневом widgets.local.json (папка ./public/config/platform/) укажите путь до собираемого пакета:

```json
{
  "widgets": {
    "packages": {
      "my-custom-widgets": {
        "url": "./widgets.local/my-custom-widgets"
      }
    }
  }
}
```

**Шаг 6.3. Деплой на сервер.** Скопируйте содержимое ./dist (либо указанной при сборке директории) на сервер в папку `/widgets.local/my-custom-widgets/`, где my-custom-widgets — имя, совпадающее с @goodt-widgets-package/my-custom-widgets (без префиксов @goodt-widgets-package/). Задеплойте ./dist/config/platform/widgets.local.json в ./widgets.local/config/widgets.local.json (либо объедините с существующей конфигурацией).

В результате на сервере должна получиться структура вида:

```
editor/
player/
config/
widgets/
widgets.local/
    ├── config/
    │   └── widgets.local.json     // обновлённый конфиг
    ├── my-custom-widgets/
    │   └── ... файлы сборки ...
```

После этого, зайдя в редактор (обычно https://yourdomain.com/editor), вы увидите свой виджет в соответствующей группе, а при добавлении на страницу — сможете работать с ним так же, как с любым стандартным.

#### 7. Регулярное обновление зависимостей

Платформа может обновляться, и, чтобы ваши виджеты работали стабильно, нужно периодически: обновлять зависимости (`npm run update:platform-deps`, `npm up`), пересобирать пакет (`npm run build`), повторно деплоить полученные артефакты.

#### 8. Устранение неполадок

1. Скрипт update:platform-deps падает с ошибкой `SyntaxError: Unexpected token '<'` — проверьте, правильно ли указан platformUrl в package.json; убедитесь, что сервер доступен, и URL действительно открывается.
2. Ошибка `ReferenceError: fetch is not defined` — используется Node.js версии ниже 18, нужно обновить Node.js.
3. `401` при установке пакетов (`npm i` или `npm up`) — проверьте корректность токена Nexus или настроек .npmrc.
